# Production Document

## 1. 게임 개요

플레이어의 크기를 키우고 줄이며 레이싱하는 게임이다.  
크기 변화는 단순한 외형 변화가 아니라, 주행 성능, 전투력, 피격 판정, 이동 경로 선택, 리스크 관리 전반에 영향을 주는 핵심 시스템이다.

게임의 목표는 단순히 빠르게 달리는 것이 아니라, 상황에 따라 몸집을 조절하며 위험을 감수하고 더 큰 보상을 노리는 플레이를 유도하는 것이다.  
즉, "고위험 플레이를 통해 스릴을 즐기는 레이싱 게임"을 지향한다.

## 2. 코어 메커니즘

### 2.1 크기 변화 기반 레이싱

- 플레이어는 주행 중 크기를 키우거나 줄일 수 있다.
- 크기 변화에 따라 캐릭터/탈것의 attributes가 실시간으로 변경된다.
- 변경되는 스탯은 UI에 즉시 표시되어야 한다.

### 2.2 크기가 커질 때

- 공격력과 HP가 증가한다.
- 다른 플레이어를 밀어내기 쉬워진다.
- 관성이 커져 방향 전환과 세밀한 제어가 어려워진다.
- 민첩도가 떨어진다.
- hitbox가 커져 장애물이나 공격에 맞기 쉬워진다.

### 2.3 크기가 작아질 때

- 공격력과 HP는 낮아진다.
- 더 민첩하게 움직일 수 있다.
- 작은 통로, 지름길, 회피 루트를 활용할 수 있다.
- 관성이 줄어 정밀한 조작이 쉬워진다.
- 대신 생존력과 직접 교전 능력이 낮아지는 하이리스크 하이리턴 상태가 된다.

### 2.4 전투 가능한 레이싱

- 단순 주행 게임이 아니라 공격이 가능한 레이싱/탈것 게임으로 설계한다.
- 공격 방식 예시:
  - 밀치기
  - 총 발사
  - 몸집 차이를 활용한 압박
- 큰 플레이어는 힘으로 밀어붙이고, 작은 플레이어는 민첩성과 경로 최적화로 역전하는 구조를 만든다.

## 3. 맵 디자인

맵은 단순 직선형 트랙이 아니라, 크기 상태와 조작 능력에 따라 선택지가 달라지는 구조를 가져야 한다.

### 3.1 공통 장애물

- 컨트롤로 피할 수는 있지만, 쉽게 피하기는 어려운 프로젝타일
- 굴러오는 돌 같은 대형 장애물
- 레이저 빔
- 낭떠러지

### 3.2 작은 플레이어에게 유리한 지형

- 작은 것만 통과할 수 있는 구멍
- 작은 것만 갈 수 있는 지름길
- 좁고 정밀한 조작이 필요한 회피 루트

### 3.3 큰 플레이어에게 유리한 지형

- 큰 플레이어만 안정적으로 건널 수 있는 도로의 갭
- 서로 평행하게 배치된 외나무다리 2개처럼, 작은 레이싱카는 외나무 다리 하나로만 달려야하지만, 큰 플레이어는 바퀴를 양쪽 다리에 걸칠 수 있어서 무게감과 안정성이 유리하게 작용하는 구조
- 우둘투둘한 지형에서 작은 플레이어는 빠졌다 튀어오를 수 있지만, 큰 플레이어는 상대적으로 밀어붙이며 지나갈 수 있는 구조

## 4. 플레이 감각 목표

- 플레이어가 크기 조절을 "언제 해야 하는가"를 계속 판단하게 만든다.
- 큰 상태는 강하지만 둔하고, 작은 상태는 약하지만 날카롭다.
- 안전한 플레이보다 계산된 위험 감수가 더 재미있도록 만든다.
- 레이싱과 전투, 회피와 지름길 선택이 동시에 얽히도록 설계한다.

## 5. 구현 플랜

### Phase 1. 기본 주행 프로토타입

- 플레이어가 혼자서 트랙을 달릴 수 있는 상태 구현
- 카메라, 이동, 회전, 충돌의 기본 루프 확립
- gimbal-lock-free rotation을 고려한 회전 시스템 도입

### Phase 2. 크기 변화 시스템

- 버튼 입력으로 크기가 바뀌는 기능 구현
- 크기 변화에 따라 속도, 가속, 선회력, HP, 공격력, hitbox 등 attributes 변경
- 현재 상태가 UI에 명확히 표시되도록 구현

### Phase 3. PVE 맵

- 프로젝타일, 굴러오는 장애물, 레이저, 낭떠러지 등 환경 위협 추가
- 작은 플레이어 전용 경로 / 큰 플레이어 전용 경로 구현
- 맵 기믹과 크기 변화의 상호작용 검증

### Phase 4. 다른 플레이어 추가

- 다른 플레이어 또는 AI 라이더 추가
- 밀치기, 충돌, 공격 상호작용 구현
- 몸집 차이에 따른 압박과 역전 구도 강화

## 6. 게임 제작 원칙

### 6.1 기술 원칙

- threejs를 활용한다.
- gimbal-lock-free rotation을 반드시 고려한다.
- 단계별 점진적 구현을 따른다.
- 반복 사용 가능하고 PCG를 통해 변조 가능한 단위 에셋을 제작한다.

### 6.2 객체 중심 제작 방식

- 각 물체는 object-oriented programming의 객체처럼 관리한다.
- 각 객체는 가능하면 parametric하게 형상을 조정할 수 있어야 한다.
- 반복 사용 가능한 구조를 우선한다.
- 물체를 구성하는 물리적으로 이어진 파트는 코드상으로도 실제로 이어진 구조가 되게 programmatic하게 구현한다.
- 서로 연결된 에셋은 joint를 쉽게 구현하고 관리할 수 있도록 계층적 구조와 연결 정보를 명시적으로 가진다.

### 6.3 좌표계 원칙

- 구현 및 검증 시 항상 좌표계를 의식한다.
- 오브젝트 좌표계, 카메라 좌표계, 월드 좌표계를 혼동하지 않는다.
- 이동, 회전, 충돌, 카메라 추적, 투사체 발사 방향 모두 어떤 좌표계 기준인지 명확해야 한다.

## 7. 검증 원칙

철저한 검증을 기본 원칙으로 한다.

### 7.1 시각적 검증

- 최종 결과물은 반드시 스크린샷 또는 렌더링으로 직접 확인한다.
- Headless Chrome 기반으로 단계별 시각적 검증을 수행한다.
- 참고 도구: https://github.com/openai/skills/tree/main/skills/.curated/playwright-interactive
- 검증 항목 예시:
  - 크기 변화가 실제 모델/충돌 범위/UI에 일관되게 반영되는가
  - 카메라 추적이 좌표계 문제 없이 안정적으로 동작하는가
  - 지름길, 구멍, 갭, 장애물 등이 의도한 크기 조건에서만 유효한가

### 7.2 수학적/논리적 검증

- interaction, 기능, 외형이 상식적이고 논리적으로 타당한지 검토한다.
- 코드상 또는 수학적으로 검토 가능한 논리는 반드시 별도 검토 스크립트로 검증한다.
- 검토 스크립트 예시:
  - 크기 배율에 따른 hitbox/질량/가속/선회력 변화가 설계 의도와 일치하는지 확인
  - 특정 크기에서 통과 가능한 구멍, 경사, 갭이 실제 수치상 성립하는지 계산
  - 투사체와 충돌 판정이 월드 좌표 기준으로 일관적인지 확인
  - 카메라 forward/right/up 벡터와 플레이어 회전값 사이에 모순이 없는지 확인

## 8. 핵심 제작 체크리스트

- 각 물체는 객체처럼 관리하고, 가능하면 파라메트릭하게 형상을 조정할 수 있게 만든다.
- 물리적으로 이어진 파트는 코드상으로도 이어진 구조가 되게 만든다.
- 오브젝트 좌표계, 카메라 좌표계, 월드 좌표계를 항상 의식한다.
- 구현은 단계적으로 진행하고, 각 단계마다 시각적/수학적 검증을 수행한다.
- 크기 변화가 재미의 중심이 되도록 레이싱, 전투, 지형, UI를 하나의 시스템으로 엮는다.
